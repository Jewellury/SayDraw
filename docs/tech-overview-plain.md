# SayDraw 技术方案大白话版

> 这是一份给非工程师看的技术速览。要工程细节请看底部"深入阅读"列出的计划文档。
> 最后更新：2026-06-20

## 一句话总结

SayDraw 现在做两件 AI 的事：**画图** 和 **听懂孩子说话**。

画图这块，我们正在把模型从 DeepSeek 换成**豆包 pro**（字节跳动的火山方舟上的大模型），让它直接写一段 SVG 代码出来，浏览器把这段代码画成"线条自己一笔一笔描出来"的动画。

语音这块，我们正在把"听懂孩子说话"从浏览器自带的语音识别换成**火山引擎的豆包流式语音识别 2.0**（也是字节家的），它对小孩子软软的、含糊的发音识别得更准。

> **状态说明（2026-06-20）：** 这两件事的完整工程计划已经写好（TASK-017 和 TASK-018），正在等你最终批准后开始执行。**目前线上的版本仍然是 DeepSeek 画图 + 浏览器自带语音**，所以下面说的"豆包 pro"和"火山 ASR"是我们正在切过去的目标方案，不是已经跑起来的代码。

## 画图（文本 → SVG 黑白线稿）

### 它在干什么

爸爸或孩子说一句话（或打字），系统把这句话连同之前的故事发给豆包 pro，让它**写一段 SVG 代码**回来。SVG 就是用文字描述画面的格式，比如 `<circle cx="50" cy="50" r="20">` 就是画一个圆。浏览器拿到这段代码后，会画出黑白的线条画面，并且用一种叫"stroke-dashoffset"的小技巧让线条**看起来像被人一笔一笔描出来的一样**——这就是 SayDraw 的灵魂动画。

**流程（文字版流程图）：**

```
用户说话/打字
   ↓
文字发给 Vercel 服务端 (/api/story/generate)
   ↓
服务端调用豆包 pro：「这是故事和最新一句，请写一段黑白 SVG」
   ↓
豆包 pro 返回一个 JSON：{ narration: "旁白", svg: "<svg>...</svg>", ... }
   ↓
服务端把 SVG 过一遍"安全过滤器"（去掉脚本、外部链接等危险内容）
   ↓
浏览器收到 SVG，用 stroke-dashoffset 动画把它一笔一笔画出来
```

### 为什么不直接用 AI 生成图片

这是 SayDraw 的**根本决定**（AGENTS.md 里叫"Sacred Product Decisions"，神圣不可改），有三条原因：

1. **灵魂是"线条自己画出来"的动画。** 如果用 AI 生成一张 PNG 图片（位图，就是一堆像素点），那个"笔尖顺着线条走"的动画效果根本做不出来——位图里没有"线条"的概念，只有色块。SVG 是用数学描述线条的，每条线都有"长度"，所以能让笔尖顺着它走。
2. **黑白线稿才像孩子的画。** SayDraw 要的是孩子随手画的那种感觉，不是彩色照片。所有线条统一用墨色 `#211e18`，不填色。
3. **SVG 是文字，好处理。** PNG 是二进制大文件，SVG 就是一段代码，能直接放进 JSON 里跟"旁白文字"一起返回，一次请求拿全。

**通俗版本的三条 Sacred Decisions：**
- **只画黑白线稿**，不要彩色插图。
- **SVG 是主输出**，不输出 PNG 或任何位图。
- **核心动画是线条自己描出来**（stroke-dashoffset），不是淡入淡出。

### 怎么切换模型

这部分用一个**"三选一自动判断"**的逻辑（TASK-017 里叫"env-preference resolver"，环境变量偏好解析器）。规则很简单：

1. 如果设了 `DOUBAO_API_KEY` → **走豆包 pro**（主力，但 2026-06-20 bakeoff 实测豆包全系太慢已注掉）
2. 如果没设豆包、但设了 `DEEPSEEK_API_KEY` → **走 DeepSeek chat（V3）**（当前实际使用）
3. 两个都没设 → **用 mock 假数据**（让孩子也能玩，只是画面是预设的）

这个设计的好处是：**万一豆包 pro 出问题（比如机房断网、key 过期），只要把豆包的 key 删掉，系统就自动退回 DeepSeek，不用改代码、不用重新部署**。2026-06-20 bakeoff 实测就是靠这个设计保住画图功能：发现豆包全系超时后，只注掉一个环境变量就切回了 DeepSeek。

### 用到什么服务

- **DeepSeek chat（V3）** —— **当前实际使用的画图模型**。2026-06-20 bakeoff 实测 0.1-0.5 秒，SVG 元素 11-16 个，无彩色。速度快、成本低。
- **豆包 pro / lite（火山方舟）** —— 预留但已注掉。实测 PRO（视觉端点）31-45s 超时、LITE 25s 超时，均远超 Vercel 30s 上限。等火山出更快的纯文本型号后可重测——改一个环境变量就行，不用改代码。
- **mock 假数据** —— 两个 key 都没配时的兜底，让孩子也能体验完整流程。

## 语音识别（说话 → 文字）

### 它在干什么

孩子按住麦克风按钮开始说话，松手后系统把这段录音送到**火山引擎的豆包流式语音识别 2.0**，它返回一段文字，文字进入输入框，然后爸爸/孩子点"画出来"按钮，进入下一步。

这里有个**关键的架构设计**：浏览器**不直接**连火山，中间要 Vercel 服务端转一道。下面专门讲为什么。

**流程（文字版流程图）：**

```
孩子按住麦克风
   ↓
浏览器用 AudioWorklet（一个浏览器音频 API）把声音录成 PCM 原始音频
   （PCM = 没有任何压缩的纯音频数据，16kHz 采样率、16 位、单声道）
   ↓
孩子松手
   ↓
浏览器把这段 PCM 音频上传到 Vercel 服务端 (/api/voice/transcribe)
   ↓
Vercel 服务端扮演"WebSocket 客户端"，去连火山的 bigmodel_nostream 接口
   （它把音频按火山的二进制协议切成小块，一块一块发给火山）
   ↓
火山识别完，返回一段文字
   ↓
Vercel 把文字回传给浏览器
   ↓
文字进入输入框
```

### 为什么中间要 Vercel 转一道（不直接浏览器连火山）

**这是整个语音方案里最关键的决策**（TASK-018 ADR-3）。原因有三层，缺一不可：

1. **浏览器的 WebSocket 不允许加自定义 HTTP 头。** WebSocket 协议天生就规定：浏览器握手时只能带几个固定的头（比如 Cookie、User-Agent），程序代码里**塞不进去**自定义的 `X-Api-Key` 之类的头。而火山的鉴权**必须**放在 `X-Api-Key`、`X-Api-Resource-Id` 这些自定义头里——这是硬性要求。
2. **就算能塞，key 也不能给浏览器。** SayDraw 的根本规则（AGENTS.md "DeepSeek API Key - Critical Rule"）是：**所有第三方 key 都只能在服务端读，永远不能暴露给浏览器**。如果浏览器直连火山，等于把火山的 key 写进前端代码里，任何打开网页的人都能在浏览器开发者工具里看到，这是绝对不允许的。
3. **服务端（Node.js）有专门的库能干这事。** Vercel 跑的是 Node.js，可以用 `ws` 这个 npm 包（Node.js 的 WebSocket 库），它**支持**在握手时加任意 HTTP 头。所以让 Vercel 当"中间人"，它拿着 key 去连火山，浏览器只跟 Vercel 用普通的 HTTP 来回——两边各自做擅长的事。

**一句话总结：浏览器不允许这样做，所以让 Vercel 帮忙。**

### 为什么失去了"边说边出字"的效果

这是当时你为了**不自建中转服务**主动接受的代价。

像豆包 App 那种"孩子一边说、字一边蹦出来"的丝滑感，技术上要求：浏览器跟服务器之间维持一条**长连接**，服务器把识别出的每个字实时推过来。要做到这一点，**必须自己搭一个一直在线的 WebSocket 中转服务**（不能是 Vercel 这种"来一次请求干一次活"的无服务器函数）。

你当时明确说了"不自建"——因为 SayDraw 是给自己家孩子用的个人工具，为一个体验细节去搭一套要长期维护、要付服务器钱的中转服务，不划算。所以选了 **`bigmodel_nostream`（一次性识别）接口**：孩子按住说完，松手，等 1-3 秒，整段文字一次性出现。

> **代价：** 孩子松手后会有一段"安静等待"的时间（通常 1-3 秒），看不到"边说边出字"的效果。
> **换来的好处：** 不用搭新服务器、不用长期维护、key 安全、代码简单。

### fallback 三级

SayDraw 的语音有三层"安全网"（TASK-018 里叫"three-tier fallback chain"），像下楼梯一样：

1. **第一层（首选）：火山豆包 ASR** —— 服务端配了 `VOLC_*` key 就用它。识别最准。
2. **第二层（退而求其次）：浏览器原生 Web Speech API** —— 火山没配 key、或者火山的探测请求失败时，自动用浏览器自带的语音识别。Chrome 上比较好用，但对孩子声音不太准。
3. **第三层（永远兜底）：打字** —— 不管上面两层怎么样，输入框永远能打字。这是 Sacred Decision #4 要求的。

**重要的小细节（TASK-018 修的一个 bug）：** 三层之间的选择发生在**按下麦克风的那一刻**，而不是松手后。因为松手后再去退到 Web Speech 已经来不及了——录音已经结束了，Web Speech 没法"回头再听一遍"。所以系统在页面打开时就去问一次 Vercel"火山能用吗？"，把答案缓存下来，孩子按下麦克风时立刻根据这个答案选路。

### 用到什么服务

- **豆包流式语音识别 2.0（火山引擎）** —— 主力。走 `bigmodel_nostream` 接口（一次性识别，准确率最高，且是唯一支持中英文 `language` 字段的接口）。资源 ID 默认是 `volc.seedasr.sauc.duration`（2.0 小时版，按识别时长计费，适合 SayDraw 这种偶尔用用的场景）。
- **浏览器原生 Web Speech API** —— 备胎。Safari 14.1 以下、某些应用内浏览器不支持 AudioWorklet 时也会退到这里。
- **打字输入** —— 永远的兜底，神圣不可删。

## 你需要提供什么 Key 和 ID

### 画图需要（TASK-017）

三个环境变量，全部**只在服务端读取**，永远不要前缀 `NEXT_PUBLIC_`：

```env
# 画图备胎 —— DeepSeek（当前实际使用的画图模型）
# 2026-06-20 bakeoff 实测 deepseek-chat 0.1-0.5s 响应，豆包全系 25-45s 超时
DEEPSEEK_API_KEY=XX
DEEPSEEK_BASE_URL=https://api.deepseek.com

# 画图主力预留 —— 豆包 via 火山方舟（已注掉，后续火山出更快模型时可解注重测）
# DOUBAO_API_KEY=XX
# DOUBAO_BASE_URL=https://ark.cn-beijing.volces.com/api/v3
# DOUBAO_MODEL=ep-your-endpoint-id
```

**判断规则：** 设了 `DOUBAO_API_KEY` 走豆包；没设豆包、设了 `DEEPSEEK_API_KEY` 走 DeepSeek；都没设用 mock 假数据。2026-06-20 bakeoff 后豆包 key 已永久注掉，当前实际上走 DeepSeek chat（V3）。

### 语音需要（TASK-018）

火山的鉴权分**新版控制台**和**老版控制台**两种形状，二选一即可（代码会自动判断）：

```env
# === 语音主力 —— 火山引擎 豆包流式语音识别 ===

# 新版控制台（推荐）：只设这一个
VOLC_API_KEY=XX

# 老版控制台：设下面两个（不要同时设 VOLC_API_KEY）
VOLC_APP_KEY=XX
VOLC_ACCESS_KEY=XX

# 下面这些都有默认值，一般不用动：
# 资源 ID —— 默认 volc.seedasr.sauc.duration（2.0 小时版）
VOLC_RESOURCE_ID=volc.seedasr.sauc.duration
# 火山 ASR 服务地址
VOLC_BASE_URL=wss://openspeech.bytedance.com
# 接口路径（一次性识别接口）
VOLC_ENDPOINT_PATH=/api/v3/sauc/bigmodel_nostream
# 模型名
VOLC_MODEL_NAME=bigmodel
```

**几个重点提醒：**

- **新版 vs 老版控制台二选一**：你的火山控制台账号要么是新版（拿到一个 `VOLC_API_KEY`），要么是老版（拿到 `VOLC_APP_KEY` 和 `VOLC_ACCESS_KEY` 两个），看你在火山控制台里实际拿到什么就配什么。
- **`VOLC_RESOURCE_ID` 默认填 `volc.seedasr.sauc.duration`**：这是 2.0 模型的小时版（按识别音频秒数计费）。你账号里 2.0 的"小时版"和"并发版"都显示运行中，但 SayDraw 就一个孩子偶尔按一下，按秒计费的小时版最划算；并发版是给高并发商业场景准备的，不适合我们。
- **不需要创建"推理接入点"**：跟画图不一样，语音这边火山的 endpoint 是**固定写死的**（`bigmodel_nostream`），不需要像画图那样去控制台创建接入点拿 `ep-xxxxx`。

### 配置文件在哪

- **`.env.local`** —— **实际生效**的配置文件，在项目根目录，**不进 git**（被 `.gitignore` 排除）。你的真实 key 只放这里。
- **`.env.example`** —— **模板**，进 git，给别人/未来的自己看"需要哪些变量"。里面全是占位符 `XX`，绝对不能写真 key。

> **提醒：** TASK-017 和 TASK-018 的 execute-agent 在执行时会负责更新 `.env.example`（把豆包和火山的变量加进去），你不用手动改这个模板文件。

## 经验教训（2026-06-20 复盘记录）

这两条是从 TASK-017 和 TASK-018 的实施过程中总结出来的，写入文档供后续项目参考：

### 1. 长任务超过 30 分钟容易失控，需要人工断点介入

**现象：** 本次两个任务中，audit-agent 和 execute-agent 的某些子任务跑到了 30 分钟以上，最终被平台超时中断。中断后任务状态残留、结果无法找回，需要人工手动指派 agent 重新运行。

**教训：**
- **拆分阶段执行** —— 单个 agent 任务控制在 5-15 分钟内。像 TASK-018 这种"先从文档研究再执行再写日志"的任务，应当拆成"研究→汇报→批准→执行→汇报→批准→审计"多个小段，每段保存结果再继续。
- **长任务要定时 checkpoint** —— 如果确实需要长时间执行（比如跑 bakeoff 测多个模型），每完成一个关键步骤就把中间结果写到文件，这样中断后可以从 checkpoint 恢复，不用重跑全部。
- **遇超时不要反复重试一模一样的指令** —— 超时通常是任务边界太大，应当精简指令、缩小范围、分段执行；反复重试会重复同样的超时。

### 2. 技术选型前对模型的调研，怎么强调都不过分

**现象：** 本次画图方案做了三次"发现-重来"循环：
1. 最初以为能用豆包 pro 替代 DeepSeek 生成 SVG（计划 v1）
2. 第一次执行时发现 `ep-20260619155735-lx4dr` 是个**视觉模型**（nana 的拍照识题 endpoint，`doubao-seed-2-0-pro-260215`），根本不适合文本生成 SVG，超时 30s+
3. 换成豆包 LITE（`doubao-seed-2-0-lite`，文本模型），实测仍然 25s——se ed-2.0 全系跑长 system prompt 都慢
4. 最后 bakeoff 实测发现 DeepSeek chat（V3）反而 0.5s 以内响应

换句话说：**选型时对模型类型（视觉 vs 文本）、速度（实测不要估算）、endpoint 映射（控制台里的 ID 对应的真实模型名）的验证，在写任何一行代码之前就该做完。**

**教训：**
- **新模型接入前，先用 10 行脚本跑一次实测**。不要依赖文档里的"理论性能"，也不要假设"豆包 pro 比豆包 lite 强就适合所有场景"——endpoint 类型（视觉 vs 推理 vs 对话）、system prompt 长度、输出格式（JSON mode）都会显著影响延迟。
- **endpoint ID 不是透明的**：`ep-xxxxx` 在控制台里可能有"用户自定义名称"，但背后映射的真实模型（`doubao-seed-2-0-pro-260215`）只有环境变量或元数据里才能看到。以后接新模型时第一步就是查清"这个 endpoint 背后的真实模型是什么"。
- **bakeoff 习惯应当固化**：任何候选模型替换，先用同一段 prompt 让所有候选跑一次，延迟和质量对比数据才是决策依据。TASK-017 的 `scripts/test-drawing-models.mjs` 就是这套方法的产物——以后类似决策可以用同一套脚本模板。

这是给复盘用的"决策索引"。每条都指向详细的 ADR（Architecture Decision Record，架构决策记录），ADR 里写了当时考虑过哪些方案、为什么没选。

- **为什么画图用豆包 pro 而不是 DeepSeek → 最终用回了 DeepSeek** —— 最初计划用豆包 pro 解决 SVG 几何不准的问题，但实测发现豆包的 PRO（视觉模型）和 LITE 都远远超时（25-45 秒，Vercel 上限 30 秒）；DeepSeek chat（V3）实测 0.09-0.55 秒，远超预期。最终决定长期用 DeepSeek chat，把豆包 key 永久注掉。看 `TASK-017` ADR-1 + ADR-6。
- **为什么画图用大模型写 SVG 而不是直接生成图片** —— 因为 Sacred Decision 规定了"线条自己描出来"的灵魂动画只能用 SVG 做，PNG 做不出来。看 `TASK-017` ADR-1 + AGENTS.md Sacred Decisions。
- **为什么画图保留 DeepSeek 作为备胎而不是删掉** —— 为了演示日不翻车：豆包万一连不上，删掉一个 key 就能退回 DeepSeek，不用改代码。看 `TASK-017` ADR-2。
- **为什么语音用火山的豆包 ASR 而不是继续用浏览器自带** —— 因为孩子声音软、含糊、远着说话，浏览器自带识别不准；豆包 ASR 是为孩子声音调过的大模型，是火山家里对孩子声音最准的。看 `TASK-018` ADR-1。
- **为什么语音用 `bigmodel_nostream` 接口** —— 因为它是火山三个接口里准确率最高、且唯一支持中英文 language 字段的；又是"一次性识别"接口，配合 Vercel 短连接架构刚好合适。看 `TASK-018` ADR-2。
- **为什么中间要 Vercel 中转而不能浏览器直连** —— **（最重要的架构决策）** 浏览器原生 WebSocket 不能加自定义 HTTP 头，而火山鉴权必须放在 `X-Api-*` 头里；而且 key 不能暴露给浏览器（项目安全规则）。所以只能让服务端代连。看 `TASK-018` ADR-3。
- **为什么没有"边说边出字"的实时效果** —— **（最重要的产品权衡）** 那种效果需要自建长期在线的中转服务，你明确拒绝了；选择了"松手后等 1-3 秒拿整段文字"的一次性识别。看 `TASK-018` ADR-4。
- **为什么用 AudioWorklet 采集音频而不是 MediaRecorder** —— 因为 MediaRecorder 在不同浏览器输出的格式不一样（Safari 是 mp4、Firefox 是 ogg、Chrome 是 webm），要转码就得加一堆音频编解码库；AudioWorklet 直接拿到原始 PCM，干净利落。看 `TASK-018` ADR-5。
