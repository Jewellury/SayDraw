# i18n & Fallback Bug Report

**Date:** 2026-06-20
**Auditor:** audit-agent
**Scope:** Task TASK-020 prereq investigation — i18n language leakage + semantic renderer fallback mismatch

---

## Bug 1: English mode — hint shows Chinese, drawing shows Chinese

### Root Cause

`lib/ai/prompts.ts:3,147` — `COMBINED_SYS` and `SEMANTIC_SYS` are Chinese-only system prompts. The generate route at `app/api/story/generate/route.ts:96-100` does not branch on `lang` when selecting the system prompt. The model receives a Chinese system prompt + English user message, and the Chinese system prompt dominates the language context, causing the model to output Chinese narration (`narration`, `followUpQuestion`, `storySummary`).

### Chain of Events

1. User sets app to English mode (`lang === 'en'`).
2. User speaks / types an English line and submits.
3. Client `page.tsx:302` passes `lang: 'en'` in the fetch body. ✅
4. Server `generate/route.ts:87` reads `lang` correctly. ✅
5. **Server `generate/route.ts:96-100` selects the system prompt — no `lang` branching:**
   - `COMBINED_SYS` is a Chinese prompt: `"你是一个儿童绘本「故事引擎」兼「插画师」..."` (`prompts.ts:3-79`)
   - `SEMANTIC_SYS` is a Chinese prompt: `"你是一个儿童绘本「分镜助手」..."` (`prompts.ts:147-206`)
6. Server `generate/route.ts:102-114` constructs a bilingual user message (English when `lang === 'en'`). ✅
7. DeepSeek receives: Chinese system prompt + English user message.
8. The Chinese system prompt sets the language context → model outputs Chinese for all text fields (`narration`, `followUpQuestion`, `storySummary`).
9. `narration` (Chinese) is stored in the scene and displayed as caption next to the SVG. User sees "Chinese captions on the image."
10. **For the hint (secondary effect):** `hint/route.ts:27` correctly selects `HINT_SYS_EN` when `lang === 'en'`. ✅ However, the `storySoFar` passed to the hint API was generated by the above Chinese-only prompt chain, so it contains Chinese text. The model receives: English system prompt (`HINT_SYS_EN`) + English wrapper text + Chinese story content. The mixed-language content can cause the model to output Chinese hints despite the English system prompt.

### Affected Files

| File | Line | Issue |
|------|------|-------|
| `lib/ai/prompts.ts` | 3-79 | `COMBINED_SYS` is Chinese-only; no `COMBINED_SYS_EN` exists |
| `lib/ai/prompts.ts` | 147-206 | `SEMANTIC_SYS` is Chinese-only; no `SEMANTIC_SYS_EN` exists |
| `app/api/story/generate/route.ts` | 96-100 | System prompt selection does not branch on `lang` |
| `app/api/story/hint/route.ts` | 29-32 | `storySoFar` may contain Chinese text from prior generate calls, tainting English hint generation |

### Recommended Fix Direction

1. **Add English versions of `COMBINED_SYS` and `SEMANTIC_SYS`** — `COMBINED_SYS_EN` and `SEMANTIC_SYS_EN` in `lib/ai/prompts.ts`, mirroring the structure but written in English with appropriate tone for a parent-child storytelling context.
2. **Branch system prompt selection on `lang`** in `generate/route.ts` lines 96-100, similar to how `hint/route.ts:27` already branches on `lang`.
3. **Add a language instruction to the existing Chinese prompts** as a belt-and-suspenders measure (e.g., "请用中文回复" in the Chinese prompt, "请用英文回复" → "Respond in English" in the English prompt).

## Bug 2: Semantic renderer shows "cat on moon" for "dad is eating watermelon"

### Root Cause

`lib/svg/semanticRenderer.ts:134-138` — when `parsed.components.length > 0`, the code calls `renderScene()` unconditionally. There is **no validation** that the returned components actually match the user's input. Since "eating" is not one of the 6 scene types and "watermelon" / "dad" are not in the 20-component library, the LLM falls back to outputs that match known patterns — defaulting to the first prominent example in the `SEMANTIC_SYS` prompt: "小猫坐在月亮上" (sitting: cat on moon).

The fallback at `generate/route.ts:139-148` (falling back to direct `COMBINED_SYS` SVG) only fires when `components.length === 0`, but in this case the LLM returns components (just wrong ones), so the guard is never triggered.

### Chain of Events

1. User speaks "dad is eating watermelon" in semantic mode.
2. Server sends `SEMANTIC_SYS` system prompt + "dad is eating watermelon" to DeepSeek.
3. The LLM needs to output:
   - **sceneType**: from 6 options — `standing, sitting, flying, fainted, interaction, sky`. None match "eating".
   - **components**: from 20-component library. None match "dad" or "watermelon".
4. **The LLM does its best with what it knows:**
   - It likely defaults to `sitting` (the first scene type with a full example in the prompt, `prompts.ts:193-194`)
   - It returns components matching that example: `moon` (support), `cat_body`, `cat_head`, `cat_eyes`, `cat_tail` (character/detail), and possibly `star` (background)
5. `generate/route.ts:129`: LLM response is parsed.
6. `generate/route.ts:134`: `parsed.components.length > 0` → **true** (LLM returned cat+moon components).
7. `generate/route.ts:135-138`: `renderScene(parsed.components, parsed.sceneType)` is called:
   - `semanticRenderer.ts:862-863`: `sceneType === "sitting"` → `isValidSceneType("sitting")` → `true` → uses `sitting`
   - `semanticRenderer.ts:865-866`: `renderSitting(components)` is called
   - Layouts moon at center, cat body/head/eyes/tail on top → "cat sitting on moon"
8. **Fallback at line 139-148 is never reached** because `components.length` was not 0.

### Key Design Gap: `semanticRenderer.ts:134-138`

```typescript
if (parsed.components.length > 0) {
  svg = renderScene(parsed.components, parsed.sceneType || undefined);
}
```

This trusts the LLM's component list blindly. There is no:
- Component-to-concept validation (does "cat + moon" make sense for "dad eating watermelon"?)
- Scene-type-to-content validation
- Similarity or confidence check

The secondary fallback (`inferSceneType` at `semanticRenderer.ts:91-104`) is also irrelevant here because the LLM provided an explicit `sceneType` that passes `isValidSceneType`.

### Affected Files

| File | Line | Issue |
|------|------|-------|
| `lib/ai/prompts.ts` | 147-206 | `SEMANTIC_SYS` has only 6 scene types (none for eating, playing, etc.) and 20 components (no dad, watermelon, etc.) |
| `lib/svg/semanticRenderer.ts` | 134-138 | No validation of LLM-returned components; trusts any non-empty component list |
| `lib/svg/semanticRenderer.ts` | 861-879 | `renderScene` default fallback is always `sitting`, matching the most common LLM hallucination pattern |
| `app/api/story/generate/route.ts` | 134-149 | Fallback to direct SVG only when `components.length === 0`; missing case where components exist but are semantically wrong |

### Recommended Fix Direction

1. **Add a semantic confidence gate**: Before calling `renderScene`, validate that the returned components make sense for the input. Options:
   - Ask the LLM to also return a confidence score — if below threshold, fall back to direct SVG
   - Check if any returned component ID is semantically related to any word in the user's input
   - Add a second quick LLM call: "Do these components {list} reasonably represent '{user input}'? Reply yes/no."
2. **Expand the fallback guard**: Change the condition at `generate/route.ts:134` from `components.length > 0` to something that also considers component relevance. Even a simple heuristic like "if the only character components are cat/dino and the user mentioned neither, fall back to direct SVG" would catch this case.
3. **Improve LLM prompting**: Add explicit instruction to `SEMANTIC_SYS`: "If the scene does not fit any of the 6 scene types, return `components: []` so the system can use an alternative renderer." This would cause the existing `components.length === 0` fallback to trigger.
4. **Medium-term**: Expand the 6 scene types and 20 components to cover more common preschool story scenarios (eating, playing, sleeping, etc.; more foods, more characters including a human/dad figure).

## Cross-Bug Note

Both bugs share a common weakness: **the server-side system prompts are language-locked to Chinese, and the 6-scene/20-component semantic library is locked to a narrow domain** (dinosaurs, cats, moons, meteors). When the user diverges from the expected Chinese+cat/dino domain, both the i18n prompt pipeline and the semantic renderer pipeline silently produce wrong output. A coordinated fix to expand prompt language support AND semantic component/scene coverage would address both root causes.
